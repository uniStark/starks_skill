# starks PM Orchestration Reference

本页定义完整档任务在用户确认方案后的执行期协议。`SKILL.md` 中的 HARD-GATE、两阶段审查与完成门禁仍是上位约束；这里不放宽任何授权或安全边界。

## 派活单与状态

PM 先把已批准范围建成 epic 与依赖 DAG，再用“派活单”承载可调度切片。每张派活单必须写清：

- ID、这活干啥、依赖与 owner；
- “随身小抄”：当前有效方案、接口约束和完成本切片所需的最小事实；
- 允许查看、允许动的地盘与禁区；
- 能力证：`read-only`、`read-write`、`execute` 或 `all`；
- 工位：共享目录或隔离 worktree；
- 应交产物、怎么算过与完成证据；
- 回话上限：默认 2000 个 Unicode 字符；
- 第几次返工、当前状态与阻塞原因。

状态使用 `Backlog`、`Ready`、`Running`、`Spec Review`、`Code Review`、`Needs Input`、`Blocked`、`Failed`、`Done`、`Dropped`。`HARD-GATE` 是执行前或范围变化后的授权门禁，不是把未批准工作伪装成可调度状态。

只有同时满足以下条件的派活单才能从 `Backlog` 进入 `Ready`：依赖已完成、写入集合可安全占用、验收标准清晰。执行完成后依次进入 `Spec Review` 与 `Code Review`；两阶段都通过且证据齐全才标 `Done`。等待用户决定标 `Needs Input`，外部条件未满足标 `Blocked`，执行已失败且没有安全后续动作标 `Failed`。取消或被替换的工作标 `Dropped`，不能计入完成数。

## 单向供料与平坦代理树

主 PM 是任务上下文的唯一汇合点。派发前由 PM 读取当前会话、已批准方案、必要项目文档、现场代码和获准记忆，再把本切片真正需要的内容压缩进“随身小抄”。不要把完整 session、长篇项目文档、旧计划、整段 commit 历史或原始记忆结果复制给子代理。

子代理遵守以下边界：

- 不主动读取或搜索 session、项目记忆、通用 README / 设计文档、近期 commit 与无关目录；
- 可读取派活单点名的目标文件、完成任务必需的直接依赖，以及平台自动注入或对目标写集合强制生效的项目规则；
- PM 应在派发前解析声明写集合适用的项目规则并放入随身小抄；若写集合需要扩大，子代理停止并回报，不自行扫描或落地；
- 上下文不足时返回“缺料”收工小票，说明缺什么、为什么需要；不得用扩大搜索范围来猜；
- 只有主 PM 可以派代理。所有 implementer、explorer 与 reviewer 都不得再派生代理，代理树最大深度为一；
- 子代理不直接访问 Obsidian 或其他共享记忆；即使主 PM 已获读取授权，也只下发与当前切片有关的必要事实。

原始日志、长 diff 和完整测试输出不塞回主上下文：收工小票给出命令、结论、关键错误摘录和产物路径。确需超过派活单的回话上限时，先写入安全的任务产物或请求 PM 放宽，不能静默倾倒全文。

## 能力证与工位

| 能力证 | 可读 | 可写 | 可执行 | 典型角色 |
|---|---:|---:|---:|---|
| `read-only` | 是 | 否 | 否 | 规格审查、代码审查、静态研究 |
| `read-write` | 是 | 是 | 否 | 无需命令的文档或代码编辑 |
| `execute` | 是 | 否 | 是 | 构建、测试、运行态验证 |
| `all` | 是 | 是 | 是 | 确需编辑并验证的实现切片 |

平台支持原生能力限制时使用原生限制；不支持时把边界写进派活单并如实说明它只是 prompt 约束，不能冒充已建立沙箱。共享目录只承载写集合互斥的任务；写集合可能相撞时串行，或使用真正隔离的 worktree。

## 收工小票

每个子代理返回以下短格式；没有合格小票，只表示代理停止，不表示任务完成：

```text
【收工小票】
- 收工状态：已交卷 / 缺料 / 等老板拍板 / 翻车
- 动了什么：
- 验收证据：
- 留下的雷：
- 产物位置：
- 建议下一棒：
```

`已交卷` 进入 `Spec Review`，不能直接标 `Done`；`缺料` 进入 `Blocked`，`等老板拍板` 进入 `Needs Input`，`翻车` 进入 `Failed`。小票缺字段、超出范围或没有证据时，PM 最多向原代理追补一次；仍不合格就标 `Failed` 或 `Blocked`，不得无限重试、换代理后假装第一次执行。

## 工作保持型调度

PM 维护 `Ready` 队列和 review 队列，并在每次派发、完成、阻塞、审查退回或用户变更后重新计算可运行派活单。

- 只要存在安全的 `Ready` 派活单且有空闲子代理槽位，立即派发补位，不等待同一波其他派活单结束。
- 调度受可用并发槽位约束；并行派活单的写入集合必须互斥。写冲突必须串行，或放入真正隔离的 worktree。
- review 工作应及时进入空闲槽位，避免实现持续占满资源而让审查饥饿；但不永久预留 reviewer 槽。没有待审卡片时，全部槽位都可用于实现。
- 没有满足依赖、写安全和验收条件的 `Ready` 卡片时，允许槽位空闲；不得用不安全派发制造“满载”。
- 不为增加子代理数量强拆任务。拆分必须形成可独立验收的产物，并降低关键路径或风险。

每个子代理 prompt 都以派活单为完整任务入口；实现代理不得兼任自己切片的 spec 或 code reviewer。

## 主 PM 与看板

主 PM 负责调度、集成、风险决策和用户通信。存在可委派任务时，主 PM 不长期占用大块实现工作，以便及时接收子代理结果和用户新消息。

执行期间在 commentary 展示真实 Markdown 看板；每次派发、完成、阻塞、审查变化或需求变化后更新，长时间任务也应定期刷新。看板至少包含：

| ID | 状态 | owner | 依赖或阻塞 | 验收或证据 |
|---|---|---|---|---|
| 示例 | Running | agent-name | dep-1 已完成 | `test-name` 待运行 |

看板汇总只报告可核验计数：`Done/Total`、`Running`、`Review`、`Needs Input`、`Blocked`、`Failed`。不得把启动、主观估计或未审查产物换算成虚假百分比，也不得编造 ETA。`Dropped` 应单列或从 Total 的口径中明确排除。

用户询问状态或追加需求时，主 PM 先回应并按下节分类；仅在对应分类动作要求时更新 DAG，`QUERY` 不改变 DAG。不需要为展示看板而打断无关的安全运行任务。

## 动态接单

收到执行期消息后，PM 先按下表分类。只有会改变调度且无法判断“追加还是替换”时，才问一个最小澄清问题。

| 代码 | 类型 | 调度动作 |
|---|---|---|
| `QUERY` | 状态、解释或问题 | 回答并刷新必要证据；不改变 DAG。 |
| `ADD` | 独立追加 | 新建派活单进入 `Backlog`；满足依赖、写安全和验收后转 `Ready`，有空闲槽位即派发。 |
| `CHANGE` | 变更现有范围 | 只暂停和重算受影响卡片，保留无关运行；若改变已批准的计划、架构或验收标准，重新打开 HARD-GATE，确认后再恢复受影响范围。 |
| `REPLACE` | 替换或取消 | 停止受影响卡片；破坏性丢弃已有成果前先确认，最终标 `Dropped`，不冒充 `Done`。 |
| `PRIORITY` | 明确插队或紧急 | 重排 `Ready` 队列；除非用户明确要求或安全边界需要，不打断无关的 `Running` 卡片。 |

新需求仍受原任务分档、授权和完成门禁约束。若它扩展出新的完整档范围，应先完成对应 HARD-GATE，而不是借已有确认越权实现。

## 大任务与降级

大任务用 epic 表示目标，用 DAG 表示依赖和关键路径；优先拆出接口、风险探针、独立实现与审查节点。调度优化目标是缩短安全关键路径和反馈周期，而不是最大化代理数。

平台槽位少或缺少隔离能力不取消工作保持型调度：仍对安全的 `Ready` / review 派活单持续补位，写冲突改为串行；即使只有一个槽位，也在当前派活单返回后立即重算队列并派发下一张派活单。

只有平台 API 或工具本身无法逐项派发、逐项等待结果时，才允许用小波次适配。任一结果返回后都必须立即重算 `Ready` / review 队列，不等待整波；若平台确实只能整批返回，须如实说明该能力限制，并在整批返回后立即重算。任何适配仍须遵守依赖、写入安全、独立验收、两阶段审查、真实看板、HARD-GATE 与完成门禁。
