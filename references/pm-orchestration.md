# starks PM Orchestration Reference

本页定义完整档任务在用户确认方案后的执行期协议。`SKILL.md` 中的 HARD-GATE、两阶段审查与完成门禁仍是上位约束；这里不放宽任何授权或安全边界。

## 任务卡与状态

PM 先把已批准范围建成 epic 与依赖 DAG，再用任务卡承载可调度切片。每张卡必须写清：

- ID 与目标；
- 依赖；
- 写入所有权（文件、目录或隔离 worktree）；
- 预期产物；
- 验收标准；
- owner；
- 当前状态；
- 阻塞原因或完成证据。

状态使用 `Backlog`、`Ready`、`Running`、`Spec Review`、`Code Review`、`Blocked`、`Done`、`Dropped`。`HARD-GATE` 是执行前或范围变化后的授权门禁，不是把未批准工作伪装成可调度状态。

只有同时满足以下条件的卡片才能从 `Backlog` 进入 `Ready`：依赖已完成、写入集合可安全占用、验收标准清晰。执行完成后依次进入 `Spec Review` 与 `Code Review`；两阶段都通过且证据齐全才标 `Done`。取消或被替换的工作标 `Dropped`，不能计入完成数。

## 工作保持型调度

PM 维护 `Ready` 队列和 review 队列，并在每次派发、完成、阻塞、审查退回或用户变更后重新计算可运行卡片。

- 只要存在安全的 `Ready` 卡片且有空闲子代理槽位，立即派发补位，不等待同一波其他卡片结束。
- 调度受可用并发槽位约束；并行卡片的写入集合必须互斥。写冲突必须串行，或放入真正隔离的 worktree。
- review 工作应及时进入空闲槽位，避免实现持续占满资源而让审查饥饿；但不永久预留 reviewer 槽。没有待审卡片时，全部槽位都可用于实现。
- 没有满足依赖、写安全和验收条件的 `Ready` 卡片时，允许槽位空闲；不得用不安全派发制造“满载”。
- 不为增加子代理数量强拆任务。拆分必须形成可独立验收的产物，并降低关键路径或风险。

每个子代理 prompt 都应 focused、self-contained，明确 focused goal、write ownership、output、acceptance 和禁止触碰的范围。实现代理不得兼任自己切片的 spec 或 code reviewer。

## 主 PM 与看板

主 PM 负责调度、集成、风险决策和用户通信。存在可委派任务时，主 PM 不长期占用大块实现工作，以便及时接收子代理结果和用户新消息。

执行期间在 commentary 展示真实 Markdown 看板；每次派发、完成、阻塞、审查变化或需求变化后更新，长时间任务也应定期刷新。看板至少包含：

| ID | 状态 | owner | 依赖或阻塞 | 验收或证据 |
|---|---|---|---|---|
| 示例 | Running | agent-name | dep-1 已完成 | `test-name` 待运行 |

看板汇总只报告可核验计数：`Done/Total`、`Running`、`Review`、`Blocked`。不得把启动、主观估计或未审查产物换算成虚假百分比，也不得编造 ETA。`Dropped` 应单列或从 Total 的口径中明确排除。

用户询问状态或追加需求时，主 PM 先回应并按下节分类；仅在对应分类动作要求时更新 DAG，`QUERY` 不改变 DAG。不需要为展示看板而打断无关的安全运行任务。

## 动态接单

收到执行期消息后，PM 先按下表分类。只有会改变调度且无法判断“追加还是替换”时，才问一个最小澄清问题。

| 代码 | 类型 | 调度动作 |
|---|---|---|
| `QUERY` | 状态、解释或问题 | 回答并刷新必要证据；不改变 DAG。 |
| `ADD` | 独立追加 | 建卡进入 `Backlog`；满足依赖、写安全和验收后转 `Ready`，有空闲槽位即派发。 |
| `CHANGE` | 变更现有范围 | 只暂停和重算受影响卡片，保留无关运行；若改变已批准的计划、架构或验收标准，重新打开 HARD-GATE，确认后再恢复受影响范围。 |
| `REPLACE` | 替换或取消 | 停止受影响卡片；破坏性丢弃已有成果前先确认，最终标 `Dropped`，不冒充 `Done`。 |
| `PRIORITY` | 明确插队或紧急 | 重排 `Ready` 队列；除非用户明确要求或安全边界需要，不打断无关的 `Running` 卡片。 |

新需求仍受原任务分档、授权和完成门禁约束。若它扩展出新的完整档范围，应先完成对应 HARD-GATE，而不是借已有确认越权实现。

## 大任务与降级

大任务用 epic 表示目标，用 DAG 表示依赖和关键路径；优先拆出接口、风险探针、独立实现与审查节点。调度优化目标是缩短安全关键路径和反馈周期，而不是最大化代理数。

小任务、平台槽位受限或缺少隔离能力时，可以退化为单代理或小波次执行；即使退化，也必须继续遵守依赖、写入安全、独立验收、两阶段审查和真实看板。任何降级都不能绕过 HARD-GATE 或完成门禁。
